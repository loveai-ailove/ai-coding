# 开发约束

适用于当前项目内新增业务模块、扩展现有模块、调整权限与数据隔离逻辑时的 AI Agent 开发约束。

## 基本原则

- 优先复用现有架构，不重复实现权限、数据范围、校验、日志、菜单体系。
- 不引入与当前项目冲突的新 ORM、权限框架或状态管理方案。
- 新功能默认必须考虑：菜单权限、按钮权限、数据范围、日志审计、类型安全、前后端一致性。
- 实现目标不只是“功能可用”，还必须“天然兼容当前权限与数据隔离体系”。

## 技术前提

- 技术栈：Next.js App Router + Prisma + MySQL。
- 权限模型：用户 -> 角色 -> 菜单/按钮权限。
- 数据范围：`SELF`、`DEPT`、`DEPT_AND_CHILD`、`ALL`。
- 当前用户最终数据范围按激活角色中的最大范围汇总。
- 普通业务查询优先走 scoped Prisma client，不绕过现有数据范围能力。

## 数据模型约束

- 需要按部门隔离的数据，模型必须包含 `deptId`。
- 需要支持“仅本人”的数据，模型必须包含 `ownerUserId` 或 `createdBy`。
- 如果同时支持“仅本人”和“按部门范围”，两个字段都必须有。
- 相关字段应建立索引，如 `@@index([deptId])`、`@@index([ownerUserId])`。
- 不允许先做业务功能、后补数据归属字段。

## 权限约束

- 菜单权限与按钮权限必须分离。
- 菜单节点表示页面访问或列表查看权限，例如 `xxx:list`。
- 按钮节点表示具体操作权限，例如 `xxx:create`、`xxx:update`、`xxx:delete`、`xxx:export`。
- 必须支持“只有查看权限，没有新增/修改/删除权限”的场景。
- 前端按钮显隐与后端接口鉴权必须同时存在，不能只做一层。

## 数据范围约束

- 必须复用现有数据范围工具与 scoped Prisma client。
- 不要在每个模块中重复手写一套部门过滤逻辑。
- 普通业务查询不要使用 `rawPrisma` 绕过数据范围。
- 新模型接入数据范围时，应明确配置其 `deptField`、`selfField`、`fallbackField`。

## 查询与写操作约束

- 列表与详情查询优先使用 scoped Prisma client。
- 更新、删除、重置等写操作前，必须先校验目标数据是否在当前用户可见范围内。
- 新增数据时，由后端自动写入归属字段，如 `deptId = currentUser.deptId`、`ownerUserId = currentUser.id`。
- 不信任前端传入的真实归属字段值。

## API 约束

- 每个 API Route 必须先鉴权，再处理业务。
- 优先使用现有鉴权工具，例如 `requireApiPermission(...)`。
- 参数校验放在独立 validator 中，不在 Route 内零散堆叠。
- 写操作尽量记录操作日志。
- 返回结构与现有接口风格保持一致。

## 页面约束

- `page.tsx` 负责服务端取数与权限判断。
- `components/...Manager.tsx` 负责交互、弹窗、表单提交。
- 页面可见不代表可编辑，编辑、删除、导出仍需单独权限点。
- 新模块上线时要同步接入菜单和按钮权限。

## 同步更新约束

新增字段、枚举、模型或权限点后，必须同步检查并更新以下层：

- Prisma schema
- 数据库结构
- Prisma generate
- TypeScript 类型
- validator
- API 入参/出参
- 页面表单与列表展示
- 权限配置
- 必要时的初始化逻辑

## 新增模块默认步骤

1. 明确数据归属方式。
2. 设计 Prisma 模型与索引。
3. 设计菜单权限与按钮权限。
4. 编写 validator。
5. 编写 API。
6. 接入 scoped Prisma 数据范围。
7. 编写页面与交互组件。
8. 接入日志。
9. 执行类型检查与构建验证。

## 禁止事项

- 不要绕过现有 RBAC 直接写死角色判断。
- 不要把数据范围逻辑散落在多个接口里重复实现。
- 不要使用 `rawPrisma` 作为常规业务查询入口。
- 不要只做前端权限控制而省略后端鉴权。
- 不要遗漏菜单权限、按钮权限或数据归属字段。
- 不要让前端决定真实数据归属值。

## 一句话准则

新增业务功能时，始终默认：权限点独立、数据归属明确、查询走 scoped Prisma、写操作先校验可见性、前后端规则一致、不绕过现有抽象。
